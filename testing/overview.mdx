The Real Clean Architecture project follows a comprehensive testing strategy that ensures reliability and maintainability across all layers. Tests are written using Kotlin Multiplatform-compatible testing frameworks and custom utilities.

## Testing Philosophy

The testing approach in this project emphasizes:

- **Layer-specific testing**: Each layer (Domain, Data, Presentation) has dedicated tests
- **Dependency isolation**: Use of test doubles and mocks to isolate units under test
- **Real implementations**: Tests use actual business logic, not mocked behavior
- **Flow testing**: Custom utilities for testing Kotlin Flows and coroutines

## Testing Framework

### Kotlin Test

All tests use `kotlin.test` for Kotlin Multiplatform compatibility:

```kotlin
import kotlin.test.*

class ExampleTest {
    @Test
    fun `EXPECT expected behavior WHEN condition`() {
        // Arrange
        val input = "test"
        
        // Act
        val result = performAction(input)
        
        // Assert
        assertEquals(expected, result)
    }
}
```

### Test Naming Convention

Tests follow a clear naming pattern:

```
`EXPECT <expected outcome> WHEN <condition>`
```

This format makes test intent immediately clear and improves readability.

## Test Structure

Tests are organized alongside their production code in Kotlin Multiplatform modules:

```
module/
├── src/
│   ├── commonMain/kotlin/      # Production code
│   └── commonTest/kotlin/      # Common tests (KMP)
├── android/
│   └── src/test/java/          # Android-specific tests
```

## Test Doubles

The project uses simple test implementations rather than mocking frameworks:

```kotlin
private class TestRepository : Repository {
    val invocations = mutableListOf<String>()
    var returnValue: Result = Result.Success
    
    override fun performAction(id: String): Result {
        invocations.add(id)
        return returnValue
    }
}
```

This approach provides:
- **Clarity**: Test doubles are easy to understand
- **Flexibility**: Full control over behavior
- **Simplicity**: No framework dependencies

## Best Practices

<AccordionGroup>
  <Accordion title="Write focused tests">
    Each test should verify a single behavior or outcome. Keep tests small and focused.
  </Accordion>
  
  <Accordion title="Use descriptive names">
    Follow the naming convention to make test intent clear without reading the code.
  </Accordion>
  
  <Accordion title="Avoid test interdependence">
    Tests should be independent and runnable in any order.
  </Accordion>
  
  <Accordion title="Test edge cases">
    Include tests for error conditions, empty states, and boundary values.
  </Accordion>
  
  <Accordion title="Use test utilities">
    Leverage the provided test utilities for consistent, maintainable tests.
  </Accordion>
</AccordionGroup>
